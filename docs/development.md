# 开发规范

## 1. 工程约定

- 业务代码统一放在 `src/baozhi_rag`，避免脚本式堆砌。
- 按职责分层组织目录，例如 `app`、`api`、`core`、`schemas`、`domain`、`services`、`infra`，避免无边界扁平化。
- 配置统一收敛到 `core/config.py`，禁止在业务代码中直接散落读取环境变量。
- 路由优先保持直接和清晰，例如 `/health/live`，在业务复杂到需要版本治理前不引入 `v1` 目录层级。
- 通用响应模型统一通过 `schemas` 目录管理，避免接口层和业务层互相泄漏内部实现。
- 当前项目采用 Python 3 命名空间包，不依赖 `__init__.py`。
- 领域实体优先放在 `domain`，业务编排优先放在 `services`，外部依赖接入放在 `infra`。

## 2. 代码规范

- 所有函数必须显式类型标注。
- 公共模块、类、函数应补充简洁 docstring，说明职责与输入输出。
- 注释优先解释“为什么”，而不是重复“做了什么”。
- 默认执行 `ruff format` 和 `ruff check --fix`，减少纯格式争论。
- 新增功能至少补一条正向测试，涉及分支时补充异常路径测试。

## 3. 提交流程

- 提交前执行：`uv run ruff check . && uv run mypy && uv run pytest`
- 提交信息必须符合 Conventional Commits。
- 推荐一个提交只做一类事情，避免把重构、功能、依赖升级混在一起。

## 4. 分支与评审建议

- `main` 保持随时可发布。
- 功能开发建议使用 `feature/<topic>`，缺陷修复使用 `fix/<topic>`。
- 合并请求描述建议包含：背景、方案、影响面、验证结果、回滚方式。

## 5. 面向金融保险场景的后续建议

- 尽早把敏感信息脱敏、审计日志、权限边界纳入设计，不要后补。
- RAG 检索链路建议提前规划可观测性字段，例如请求编号、用户会话编号、召回文档编号。
- 回答生成应预留风控开关，避免模型直接输出高风险承诺类表述。
- `infra/llm`、`infra/retrieval`、`infra/storage` 分别承载模型、检索、存储适配，不要把外部 SDK 调用散落到 `services` 和 `api`。
